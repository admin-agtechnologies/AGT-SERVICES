\section{Introduction}

Ce document constitue le cahier des charges du \textbf{Service Chatbot IA Centralisé} (ci-après \og Chatbot Service \fg{}) destiné à l'écosystème de plateformes AGT. Il sert de référence unique pour la conception, le développement et l'intégration avec les autres services.

\subsection*{Périmètre}

Le Chatbot Service est le \textbf{cerveau IA conversationnel multi-provider} de l'écosystème. Il gère l'ensemble du cycle de traitement des messages : détection d'intention, navigation dans des flux conversationnels, génération de réponses par IA, gestion de la mémoire de conversation et déclenchement d'actions métier. Il est conçu comme un service \textbf{générique, configurable et multi-canal}, sans logique métier codée en dur. Chaque plateforme configure ses propres bots.

\begin{infobox}
\textbf{Modes d'utilisation :} Le service fonctionne (1) via le Hub Engine (AGT-Bot $\rightarrow$ WhatsApp), (2) en standalone (AGT-Market, SALMA, applications web), et (3) via connecteur externe (agent vocal, Messenger, Telegram, SMS, etc.).
\end{infobox}

\begin{warnbox}
\textbf{Nouveautés v1.1 $\rightarrow$ v1.2 :} Conversation Orchestrator formalisé comme composant central, Action System standardisé (contrat, retry, timeout, auth), frontière avec Search Service documentée (vector search interne justifié), statut \textit{implementation-ready}.
\end{warnbox}

\subsection{Contexte}

L'organisation développe plusieurs plateformes indépendantes nécessitant des capacités conversationnelles intelligentes :

\begin{tabularx}{\textwidth}{l l X}

\rowcolor{aiprimary}

AGT-Market & En développement & Assistant achat, support vendeur \\
SALMA & Migration prévue & Assistant conversationnel intégré \\
Futures & Planifié & Capacités IA à la demande \\
\bottomrule

\subsection{Problématique}

Sans service centralisé, chaque plateforme implémenterait sa propre couche IA, entraînant : duplication des intégrations avec les providers IA (coûts \& maintenance), incohérence de l'expérience conversationnelle, impossibilité de mutualiser les bases de connaissances, et multiplication des budgets IA sans contrôle.

\section{Vision \& Objectifs}

\subsection{Vision}

Un service IA conversationnel unique, générique et configurable, permettant à toute plateforme de l'écosystème de déployer des chatbots intelligents en quelques minutes, sans développement spécifique.

\section{Architecture Technique}

\subsection{Vue d'ensemble}

Le Chatbot Service est un microservice REST stateless, déployé en conteneur Docker. Il expose une API JSON consommée par les plateformes clientes (via Hub Engine ou en direct) et interagit avec les providers IA externes.

\subsection{Architecture IA Multi-Couches — Pipeline de traitement}

\begin{infobox}
\textbf{Pipeline séquentiel :} Message $\rightarrow$ \textbf{Couche 1} (mots-clés) $\rightarrow$ \textbf{Couche 2} (flux) $\rightarrow$ \textbf{Couche 3} (IA générative) $\rightarrow$ \textbf{Couche 4} (fallback). Chaque couche peut court-circuiter le pipeline si le score de confiance dépasse le seuil configuré.
\end{infobox}

\subsection{Composant Central : Conversation Orchestrator}

Le \textbf{Conversation Orchestrator} est le composant interne qui pilote le cycle de vie complet d'un message entrant. Il n'est pas un microservice séparé : c'est une classe/module Python central dans le codebase du Chatbot Service.

\subsubsection{Responsabilités}

\rowcolors{2}{aitable}{white}
\begin{tabularx}{\textwidth}{l X}
\toprule
\rowcolor{aiprimary}
\textbf{\color{white}Responsabilité} & \textbf{\color{white}Détail} \\
\midrule
Channel Validation & Vérifie que le canal est actif pour le bot, charge la config canal (adapter) \\
Context Loading & Charge le contexte conversation depuis Redis (historique, variables, compteur fallbacks) \\
Language Detection & Détecte la langue du message si non fournie \\
Pipeline Routing & Exécute séquentiellement les couches 1 $\rightarrow$ 2 $\rightarrow$ 3 $\rightarrow$ 4, court-circuite dès confiance suffisante \\
Provider Selection & Pour la couche 3, sélectionne le provider selon \code{purpose} + \code{priority}, applique le circuit breaker \\
Action Execution & Si une couche ou un flow déclenche une action, exécute via l'Action Executor \\
Transfer Decision & Si \code{consecutive\_fallbacks} $\geq$ \code{human\_transfer\_after}, déclenche le transfert S2S vers Chat \\
Response Adaptation & Adapte la réponse au canal (taille, format, médias) via le Channel Adapter \\
Context Saving & Sauvegarde le contexte mis à jour dans Redis \\
Logging & Écrit de manière asynchrone dans \code{conversation\_logs} (fire-and-forget via Celery) \\
\bottomrule
\end{tabularx}

\subsubsection{Flux d'exécution}

\begin{enumerate}[leftmargin=2em]
\item \code{POST /chatbot/converse} reçu par le contrôleur Django
\item Le contrôleur instancie l'\code{Orchestrator} avec le payload
\item L'Orchestrator charge le bot, valide le canal, charge le contexte Redis
\item L'Orchestrator exécute le pipeline :
  \begin{enumerate}
  \item \textbf{Couche 1} — \code{IntentResolver} : matching mots-clés $\rightarrow$ si confiance $\geq$ seuil, retourne
  \item \textbf{Couche 2} — \code{FlowEngine} : si un flow est actif ou déclenché par intent, exécute les nœuds (avec branching conditionnel)
  \item \textbf{Couche 3} — \code{AiEngine} : sélectionne le provider (\code{purpose=conversation}), construit le prompt (system\_prompt + KB context + historique), appelle l'API. Si échec, tente le fallback provider (\code{priority=1})
  \item \textbf{Couche 4} — \code{FallbackHandler} : retourne le \code{fallback\_message}, incrémente le compteur, évalue le transfert humain
  \end{enumerate}
\item L'Orchestrator adapte la réponse via le \code{ChannelAdapter}
\item L'Orchestrator sauvegarde le contexte dans Redis, enqueue le log async
\item Le contrôleur retourne la réponse JSON
\end{enumerate}

\begin{warnbox}
\textbf{Règle :} L'Orchestrator ne contient AUCUNE logique métier. Il orchestre des composants configurables. Toute la ``connaissance'' vient de la base de données (intentions, flows, prompts, actions).
\end{warnbox}

\subsection{Action System — Contrat Standardisé}

Toute action déclenchée par un flow node ou par le bot (table \code{bot\_actions}, \code{flow\_actions}) suit un contrat standardisé exécuté par l'\textbf{Action Executor}, un sous-composant de l'Orchestrator.

\subsubsection{Contrat d'une Action}

\begin{lstlisting}
{
  "type": "api_call",
  "service": "market",
  "endpoint": "/api/v1/orders",
  "method": "POST",
  "auth": "s2s",
  "headers": {"X-Bot-Id": "{{bot_id}}"},
  "payload_template": {
    "user_id": "{{sender_id}}",
    "product_id": "{{flow_var.product_id}}"
  },
  "timeout_ms": 3000,
  "retry": {
    "enabled": true,
    "max_attempts": 2,
    "backoff_ms": 500
  },
  "on_failure": "continue"
}
\end{lstlisting}

\subsubsection{Types d'actions supportés}

\rowcolors{2}{aitable}{white}
\begin{tabularx}{\textwidth}{l l X}
\toprule
\rowcolor{aiprimary}
\textbf{\color{white}Type} & \textbf{\color{white}Auth} & \textbf{\color{white}Description} \\
\midrule
\code{api\_call} & S2S / Bearer & Appel REST vers un service interne AGT \\
\code{webhook} & Custom headers & Appel HTTP vers un endpoint externe \\
\code{transfer\_human} & S2S & Déclenche un transfert vers le Service Chat \\
\code{send\_notification} & S2S & Envoie une notification via le Service Notification \\
\code{set\_variable} & — & Modifie une variable de session (Redis) \\
\bottomrule
\end{tabularx}

\subsubsection{Règles d'exécution}

\begin{itemize}
\item \textbf{Auth S2S :} utilise le \code{client\_id} / \code{client\_secret} du Chatbot Service enregistré dans Auth v2.1 (endpoint \code{POST /auth/s2s/token}).
\item \textbf{Template variables :} les \code{\{\{...\}\}} sont résolus au runtime depuis le contexte (sender\_id, bot\_id, flow variables, session variables).
\item \textbf{Timeout :} défaut 3000ms, configurable par action.
\item \textbf{Retry :} optionnel, exponential backoff, max 3 tentatives.
\item \textbf{on\_failure :} \code{continue} (ignore et poursuit le flow) ou \code{abort} (retourne une erreur au user).
\item \textbf{Logging :} chaque exécution d'action est loggée (succès/échec, durée, HTTP status).
\end{itemize}

\subsection{Frontière avec le Service Recherche}

\begin{warnbox}
\textbf{Décision d'architecture :} Le Chatbot Service possède sa \textbf{propre} capacité de vector search (pgvector) pour la knowledge base interne des bots. Il n'utilise PAS le Service Recherche v1.2 pour cela.
\end{warnbox}

\subsubsection{Justification}

\rowcolors{2}{aitable}{white}
\begin{tabularx}{\textwidth}{l X X}
\toprule
\rowcolor{aiprimary}
\textbf{\color{white}Critère} & \textbf{\color{white}Service Recherche v1.2} & \textbf{\color{white}Chatbot KB (pgvector)} \\
\midrule
Périmètre & Données métier plateformes (produits, FAQ, formations) & Knowledge base privée par bot (RAG) \\
Technologie & Elasticsearch (full-text + futur dense\_vector) & pgvector (similarité cosinus native PostgreSQL) \\
Latence cible & $<$ 200ms & $<$ 100ms (critique pour pipeline temps réel) \\
Cycle de vie & Index gérés par les plateformes & Embeddings gérés par le pipeline d'ingestion interne \\
Sémantique MVP & Full-text + scoring (embeddings = roadmap future) & Vector search = MVP day 1 \\
\bottomrule
\end{tabularx}

\subsubsection{Point d'intégration futur}

Quand le Service Recherche v1.2+ implémentera la recherche sémantique par embeddings (dense\_vector Elasticsearch), le Chatbot Service pourra optionnellement déléguer la recherche KB via un adapter :

\begin{itemize}
\item \code{search\_backend: "internal"} $\rightarrow$ pgvector (défaut, MVP)
\item \code{search\_backend: "search\_service"} $\rightarrow$ appel REST vers Search Service
\end{itemize}

Ce choix sera configurable par bot via \code{bot\_configs} (\code{config\_key: "search\_backend"}).

\subsection{Workers Asynchrones (Celery)}

Le service utilise des workers Celery pour les tâches lourdes ou périodiques :

\rowcolors{2}{aitable}{white}
\begin{tabularx}{\textwidth}{l l X}
\toprule
\rowcolor{aiprimary}
\textbf{\color{white}Worker} & \textbf{\color{white}Déclencheur} & \textbf{\color{white}Rôle} \\
\midrule
\code{ingest\_document} & Event (upload doc) & Extraction texte $\rightarrow$ chunking $\rightarrow$ embedding $\rightarrow$ insertion KB \\
\code{aggregate\_stats} & Cron (toutes les heures) & Agrège \code{conversation\_logs} $\rightarrow$ insère/met à jour \code{bot\_stats} \\
\code{health\_check\_providers} & Cron (toutes les 5 min) & Ping les providers IA, met à jour circuit breaker \\
\code{cleanup\_redis\_sessions} & Cron (quotidien) & Purge les sessions expirées dans Redis \\
\bottomrule
\end{tabularx}

\subsection{Stratégie de Persistance}

\begin{infobox}
\textbf{Séparation mémoire / analytique :}
\begin{itemize}
\item \textbf{Redis} (TTL configurable, défaut 30 min) : contexte conversation en cours, variables de session, compteur fallbacks consécutifs. Pas de persistance long terme.
\item \textbf{PostgreSQL} \code{conversation\_logs} : chaque échange est loggé de manière asynchrone (fire-and-forget) pour les analytics, le dashboard stats, et le dataset de fine-tuning.
\item \textbf{PostgreSQL} \code{bot\_stats} : agrégation horaire par le worker \code{aggregate\_stats} à partir des \code{conversation\_logs}.
\end{itemize}
\end{infobox}

% ── CONVERSATION ──
\subsection{Conversation — Endpoint Principal}

\begin{apibox}{POST /chatbot/converse}
\textbf{Description :} Endpoint principal. Envoie un message et reçoit la réponse du bot après traitement par le pipeline multi-couches. Le champ \code{channel} est obligatoire et détermine l'adapter de sortie.\\[4pt]
\textbf{Request body :}
\begin{lstlisting}
{
  "bot_id": "uuid",
  "conversation_id": "uuid-or-null",
  "sender_id": "uuid",
  "channel": "whatsapp",
  "message": "Bonjour, je cherche un produit",
  "language": "fr",
  "metadata": {
    "platform": "agt-bot",
    "source_message_id": "wa-msg-123"
  }
}
\end{lstlisting}
\textbf{Response 200 :}
\begin{lstlisting}
{
  "response": "Bonjour ! Quel type de produit recherchez-vous ?",
  "conversation_id": "uuid",
  "intent_detected": "greeting",
  "layer_used": "layer_1_keywords",
  "confidence": 0.88,
  "provider_used": null,
  "action": null,
  "language": "fr",
  "transfer": null,
  "metadata": {
    "processing_time_ms": 145,
    "fallback": false,
    "consecutive_fallbacks": 0
  }
}
\end{lstlisting}
\textbf{Valeurs possibles de \code{layer\_used} :}
\begin{itemize}
\item \code{layer\_1\_keywords} — Résolu par mots-clés / intentions
\item \code{layer\_2\_flow} — Résolu par flux conversationnel
\item \code{layer\_3\_ai} — Résolu par IA générative
\item \code{layer\_4\_fallback} — Fallback (aucune couche n'a résolu)
\end{itemize}
\textbf{Si transfert humain déclenché}, le champ \code{transfer} contient :
\begin{lstlisting}
{
  "transfer": {
    "id": "uuid",
    "status": "sent",
    "reason": "max_fallbacks",
    "chat_transfer_id": "uuid"
  }
}
\end{lstlisting}
\textbf{Codes :} 200 OK, 400 Validation error, 401 Unauthorized, 404 Bot not found, 429 Rate limited.
\end{apibox}

\begin{apibox}{DELETE /chatbot/conversations/\{conversationId\}}
\textbf{Description :} Suppression du contexte d'une conversation (mémoire Redis).\\
\textbf{Codes :} 200 OK, 404 Not found.
\end{apibox}

\section*{Annexes — Cas d'Usage}

\subsection*{Cas 1 : WhatsApp via Hub Engine}

\begin{enumerate}
\item L'utilisateur envoie un message WhatsApp.
\item AGT-Bot (Hub Engine) reçoit le message et appelle \code{POST /chatbot/converse} avec \code{channel: "whatsapp"}.
\item Le pipeline vérifie que le canal \code{whatsapp} est actif pour ce bot, applique l'adapter de sortie (limite taille, format texte).
\item Le pipeline multi-couches traite le message.
\item La réponse est retournée au Hub Engine qui l'envoie via WhatsApp.
\end{enumerate}

\subsection*{Cas 2 : Flux de commande (AGT-Market)}

\begin{enumerate}
\item L'utilisateur dit \og je veux commander \fg{} via le widget web (\code{channel: "web"}).
\item Couche 1 détecte l'intention \code{order}.
\item Couche 2 active le flux \og Flux commande \fg{}.
\item Le bot guide l'utilisateur à travers les nœuds (produit, quantité, confirmation).
\item Un nœud \code{condition} vérifie la disponibilité via \code{branches} (si oui $\rightarrow$ confirmation, si non $\rightarrow$ suggestion).
\item Un nœud \code{action} appelle l'API de création de commande.
\end{enumerate}

\subsection*{Cas 3 : Fallback puis transfert humain}

\begin{enumerate}
\item Le bot ne comprend pas le message (couches 1, 2, 3 échouent).
\item Couche 4 envoie le message de fallback. Le compteur \code{consecutive\_fallbacks} s'incrémente.
\item Le message est loggé dans \code{conversation\_logs} (persistance PostgreSQL).
\item Après 3 échecs consécutifs (\code{human\_transfer\_after}), le bot appelle le Service Chat en S2S : \code{POST /api/v1/chat/conversations/transfer} avec \code{bot\_history} et \code{context}.
\item Un \code{transfer\_log} est créé côté chatbot avec le statut \code{sent}.
\item Quand l'opérateur clôture, le Service Chat appelle \code{POST /chatbot/transfers/\{id\}/callback} avec \code{status: "cloture"}.
\item Le bot reprend la conversation.
\end{enumerate}

\subsection*{Cas 4 : Multi-langue}

\begin{enumerate}
\item L'utilisateur envoie un message en anglais.
\item Le service détecte la langue (EN).
\item Le pipeline utilise les intentions et prompts adaptés à la langue.
\item La réponse est générée en anglais (le provider de \code{purpose: "conversation"} est utilisé).
\end{enumerate}

\subsection*{Cas 5 : Agent vocal externe}

\begin{enumerate}
\item Un service d'agent vocal externe appelle \code{POST /chatbot/converse} avec \code{channel: "vocal"} et un Bearer token valide.
\item L'adapter vocal formate la réponse (texte court, sans markdown, compatible TTS).
\item La réponse est retournée au service vocal qui la synthétise en audio.
\end{enumerate}

\subsection*{Cas 6 : Ingestion d'un document PDF}

\begin{enumerate}
\item L'admin uploade un PDF via \code{POST /bots/\{id\}/knowledge/documents}.
\item Le fichier est stocké via le Média Service.
\item Un job Celery est créé (\code{ingestion\_jobs}, statut \code{pending}).
\item Le worker extrait le texte, découpe en chunks, génère les embeddings via le provider de \code{purpose: "rag"}.
\item Les chunks sont insérés dans \code{knowledge\_base\_entries} avec leurs embeddings.
\item Le statut du job passe à \code{done}.
\end{enumerate}

\subsection*{Cas 7 : Failover multi-provider}

\begin{enumerate}
\item Un message arrive en couche 3 (IA générative).
\item Le provider principal (\code{priority: 0}, \code{purpose: "conversation"}) est appelé.
\item L'appel échoue (timeout, erreur 500). Le circuit breaker incrémente le compteur.
\item Le service bascule automatiquement sur le provider de \code{priority: 1} (fallback).
\item Si le provider principal atteint \code{circuit\_breaker\_threshold} erreurs consécutives, il est temporairement désactivé.
\end{enumerate}

\section*{Conclusion Générale}

Le Service Chatbot IA Centralisé v1.2 est le pilier conversationnel stratégique de l'écosystème AGT. Son architecture repose sur un \textbf{Conversation Orchestrator} explicite qui pilote un pipeline 4 couches, orchestre des providers IA multiples avec failover, adapte les réponses par canal, et exécute des actions standardisées vers les services internes.

Les décisions structurantes de cette version :

\begin{itemize}
\item \textbf{Multi-provider} avec routing par \code{purpose} et circuit breaker — aucune dépendance à un fournisseur unique.
\item \textbf{Multi-canal} via adapter pattern — WhatsApp, web, vocal, SMS, API extensible sans toucher au pipeline.
\item \textbf{Action System standardisé} — contrat unique (auth, retry, timeout, template variables) pour toutes les interactions inter-services.
\item \textbf{Persistance duale} — Redis pour la mémoire temps réel, PostgreSQL pour les analytics et le fine-tuning.
\item \textbf{Frontière Search Service} documentée — pgvector interne pour le RAG MVP, adapter configurable vers Search Service quand celui-ci supportera les embeddings.
\item \textbf{Transfert humain bidirectionnel} — intégration S2S complète avec le Service Chat v1.2.
\end{itemize}

Prochaines étapes : validation équipe, initialisation des développements par priorité, déploiement progressif Hub Engine (AGT-Bot) puis extension aux autres canaux.

\section*{Annexes — Bonus / Stratégies Avancées}

\subsection*{Suggestion automatique des questions non répondues}

Le service collecte les messages passés en fallback et les agrège par similarité. L'endpoint \code{GET /bots/\{id\}/stats/unanswered} permet aux administrateurs de transformer ces messages en nouvelles intentions ou entrées de knowledge base.

\subsection*{Gestion du budget IA par bot}

Chaque bot peut avoir un budget mensuel configurable (en tokens ou en montant). Le service tracked la consommation et peut désactiver la couche 3 si le budget est épuisé, basculant directement vers le fallback.

\subsection*{A/B Testing des réponses}

Possibilité de configurer plusieurs réponses pour une même intention avec distribution pondérée, permettant de mesurer laquelle génère le meilleur engagement.

\subsection*{Stratégie de fine-tuning}

Les conversations réussies (haute confiance, pas de fallback) peuvent être exportées pour constituer un dataset de fine-tuning, améliorant progressivement les performances de la couche IA.
